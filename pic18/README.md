# 本物の PIC18 版「RTOS もどき」(gpsim で動く)

Python の教材 VM（一つ上のディレクトリ）と同じ協調型ラウンドロビンを、
**本物の PIC18 機械語**で実装したもの。gpasm でアセンブルし gpsim で走る。

## なぜ PIC でわざわざやるのか

戻り番地は PIC では「ハードウェアリターンスタック」に積まれ、深さに上限がある
（PIC18 で 31 段）。通常 RAM 上のソフトウェアスタックのように SP 一つで丸ごと
切り替えられないので、文脈切り替えのたびに

- 走行中タスクの戻り番地を `TOSU:TOSH:TOSL` から TCB へ退避し、
- 次タスクの戻り番地を積み直す

必要がある。`os_yield`（`rtos18.asm`）がこれを実際にやっている。

## 動かす

```sh
sudo apt install gputils gpsim   # どちらも OSS
make        # rtos18.asm をアセンブル
make run    # gpsim で実行し、レジスタを観察
```

## 何を見れば「動いた」と言えるか

`make run` の出力（cycle ごとのチェックポイント）:

| cycle | COUNT0 (task0:+1) | COUNT1 (task1:+2) | STKPTR |
|------:|------------------:|------------------:|-------:|
| 300   | 2                 | 4                 | 0      |
| 1500  | 10                | 20                | 1      |
| 4000  | 28                | 54                | 0      |

- **COUNT1 ≈ 2×COUNT0** … 2 つのタスクが交互に、毎回“続き”から再開している
  （＝文脈切り替えが効いている）。
- **STKPTR が 0〜1 から増えない** … タスクが無限ループで `rcall` し続けても、
  31 段のハードスタックは食い潰されない。各タスクの戻り番地は TCB に逃がして
  あるから。これが「PIC で RTOS を書くと何が特別なのか」の答え。

## 落とし穴デモ: 31 段の壁にぶつける

```sh
make run-overflow
```

`overflow.asm` は逆に「キリのいい所で yield せず、戻りもせず、ひたすら深く
`rcall` し続けるダメなタスク」を動かす。戻り番地が TCB に逃げないので、そのまま
ハードスタックに積み上がり、PIC18 の上限 31 段にぶつかる。

| cycle | depth（コードが信じる段数） | STKPTR | 意味 |
|------:|----------------------------:|:------:|------|
| 40    | 12                          | `0x0d` (13) | `depth+1` で一緒に増える |
| 110   | 36                          | `0x9f`      | = STKFUL(0x80) ＋ 31。**31 段で頭打ち＋あふれ検出** |
| 600   | 199                         | `0x9f`      | depth は増え続け、実スタックは 31 で凍結 |

`STKPTR` の下位 5 bit が段数、bit7 が `STKFUL`（あふれ検出）。`0x9f = 0x80 | 0x1f`。
`STKPTR` は最初 `depth+1` で一緒に増えるが、下位 5 bit が **31 で頭打ち＋STKFUL 点灯**。
以後は `depth` だけが増え、その差が**取りこぼした戻り番地の数＝もう正しく戻れない深さ**。
`make run`（STKPTR が 0〜1 のまま）と並べると、文脈切替で TCB に逃がす意味が一目で分かる。

> `CONFIG STVR = OFF` にしてあるので、あふれてもリセットせず STKFUL を観察できる。
> `STVR = ON` にすると、32 個目の CALL/PUSH でオーバーフローが検出され、同じバグが
> 「即リセット」へ化ける。

## ファイル

- `rtos18.asm` … カーネル（`os_start` / `os_yield` / `point_fsr0`）＋ 2 タスク
- `overflow.asm` … 落とし穴デモ（ハードスタックを溢れさせる）
- `run.stc` / `overflow.stc` … gpsim バッチスクリプト（レジスタを表示）
- `Makefile`   … アセンブル & 実行

## いまの割り切り（最小マイルストーン）

- 協調型のみ（プリエンプションなし）。タスクはキリのいい所で `rcall os_yield`。
- yield 時にハードスタックは最上位 1 段だけ、という前提（深い入れ子の途中で
  yield しない）。この単純さが、上の STKPTR の話を一番きれいに見せる。
- 退避するコンテキストは戻り番地＋W＋STATUS の最小限（単一バンク）。
- タスクは `FSR0` / `PROD` / `BSR` をまたいで使わない前提（`point_fsr0` が
  `FSR0`・`PRODL/H` を壊し、`BSR` は退避しない）。汎用の文脈切替ではなく最小デモ。
